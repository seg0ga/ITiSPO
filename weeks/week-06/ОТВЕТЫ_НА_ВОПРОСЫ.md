# Вопросы для самопроверки

1. **Зачем нужны `variables` в GraphQL запросе? Почему нельзя просто склеивать строки?**
    
    variables нужны для защиты от иньекции. Если просто склеивать строки, то пользователь может подставить такое значение, которое сломает запрос и украдет данные. К тому же при склеивании сервер неможет кэшировать разобранные запрос и каждый раз парсит его заново, а это лишняя нагрузка
2. **Из каких полей состоит стандартный JSON-запрос к GraphQL серверу?**
    
    Стандартный JSON запрос состоит минимум из 1 поля: query (строка с запросом). Но так же может быть еще 2 необязательных поля: variables (для передачи переменных) и operationName(строка, которая говорит какую именно операцию необходимо выполнить(если в query указано несколько))
3. **Почему у GraphQL обычно только один URL endpoint?**
        
    Потому что GraphQL использует единую точку входа. Вся логика того, какие данные нужны описываются внутри запроса, а не в URL. Сервер принимает запрос, разбирает его и направляет к нужным обработчикам.
4. **Что означает наличие поля `errors` в ответе сервера? Может ли оно прийти вместе с `data`?**
    
    Наличие поля errors означает то, что при выполнении запроса произошли ошибки, оно может прийти вместе с data, если сервер смог частично получить часть данных, но при получении другой части произошла ошибка
5. **Как клиент должен обрабатывать ошибки, если HTTP статус ответа 200?**
    
    Клиент не должен доверять http статусу, он должен проверять тело ответа на наличие поля errors. Если оно есть, то обрабатывать ошибки, тк graphql часто отвечает 200 при любых ошибках уровня приложения
6. **В чем плюсы использования типизированных клиентов (например, Apollo Client или Relay) перед простыми HTTP-запросами?**

    Они кэшируют данные, обновляют интерфейс при изменениях и предоставляют удобные инструменты для отладки и разработки
